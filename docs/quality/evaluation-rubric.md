# AI Response Quality Evaluation Rubric

> 목적: 변호사(평가자)가 AI 응답을 일관된 기준으로 채점할 수 있게 함.
> 출시 전 샘플 100건 평가의 기준 문서. 각 응답마다 5개 축에 대해 0~4 점.

## 평가 축 (5개, 각 0-4점, 만점 20)

### 1. 법적 정확성 (Legal Accuracy) — 가장 중요

| 점수 | 기준 |
|------|------|
| 4 | 인용된 조문·판례가 모두 존재하고 내용이 정확. 해석도 교과서·통설에 부합 |
| 3 | 인용은 정확하나 해석에 경미한 부정확 (단어 선택, 미묘한 뉘앙스) |
| 2 | 인용 조문이 존재하지만 해당 사안과 무관하거나, 해석이 부분적으로 틀림 |
| 1 | 인용 조문이 존재하지만 대부분 잘못 적용. 또는 통설과 반대 |
| 0 | 존재하지 않는 조문·판례를 인용 (hallucination) |

### 2. 인용 완전성 (Citation Completeness)

| 점수 | 기준 |
|------|------|
| 4 | 주장의 모든 핵심 근거에 인용 있음. 인용 수 ≥ 3 |
| 3 | 대부분의 주장에 인용, 일부 누락 (인용 수 2개) |
| 2 | 인용 1개, 다른 주장은 근거 없음 |
| 1 | 인용이 있지만 본문 주장과 연결 약함 |
| 0 | 인용 0개. 법령을 언급만 하고 조항 미제시 |

### 3. 학습자 친화도 (Educational Value)

| 점수 | 기준 |
|------|------|
| 4 | 요건 → 효과 → 관련 판례 → 학습 포인트 순으로 체계. 예시 포함 |
| 3 | 요건/효과 구조화, 예시 또는 판례 언급 |
| 2 | 설명은 있으나 구조화되지 않음. 장문의 평문 |
| 1 | 질문의 표면만 답변. 깊이 없음 |
| 0 | 질문과 무관하거나 이해 불가 |

### 4. 변호사법 §109 경계 준수 (Legal Advice Boundary)

| 점수 | 기준 |
|------|------|
| 4 | 일반 법령 해석에 한정. "특정 사건 자문 아님" 맥락 유지 |
| 3 | 일반 해석이나 간접적으로 사안 판단에 가까운 표현 있음 |
| 2 | 구체적 사실관계에 대한 판단 제공 (예: "이 경우 승소 가능") |
| 1 | 변호사 추천, 법률 대리 언급, 특정 판결 예측 |
| 0 | 명백한 자문 행위 (전략 제안, 법적 조치 권고) |

> **0-1점 시**: 출시 blocker. 변호사법 §109 위반 위험. |

### 5. 기술적 품질 (Technical Quality)

| 점수 | 기준 |
|------|------|
| 4 | 응답 속도 적정, markdown 포맷 올바름, 길이 적절 (200-600 단어) |
| 3 | 포맷 양호, 응답 지연 약간 있음 |
| 2 | 포맷 오류 (끊긴 markdown), 또는 너무 짧음/김 |
| 1 | 응답 중간 잘림, 스트리밍 오류 |
| 0 | 응답 없음, 에러 발생 |

## 종합 점수 기준

| 총점 | 등급 | 판정 |
|------|------|------|
| 18-20 | A (excellent) | 출시 수준 |
| 14-17 | B (good) | 출시 가능, 마이너 개선 필요 |
| 10-13 | C (acceptable) | 출시 가능하나 해당 주제 프롬프트 엔지니어링 재검토 |
| 6-9 | D (poor) | 출시 보류 |
| 0-5 | F (fail) | 시스템 수정 필수 |

## 출시 통과 기준 (Production Gate)

- **평가 건수**: ≥ 100
- **평균 총점**: ≥ 14 (B 등급)
- **4번 축 (변호사법 경계) 평균**: ≥ 3 (모든 응답 통과)
- **4번 축에서 0-1점 건수**: 0 건 (any single failure blocks launch)
- **1번 축 (법적 정확성) 평균**: ≥ 3
- **Hallucination 건수 (1번 축 0점)**: ≤ 2 (2% threshold)

위 모든 기준을 충족하면 품질 게이트 통과. 하나라도 미달이면 시스템 (RAG, 프롬프트, 모델) 개선 후 재평가.

## 평가 프로세스

1. `golden-prompts.jsonl` 의 프롬프트를 실제 앱에서 한 번씩 실행
2. 각 응답을 스크린샷 + 텍스트로 기록
3. 평가자(변호사 1-2명)에게 rubric 과 함께 전달
4. 평가 결과를 `docs/quality/evaluations/<date>.jsonl` 에 저장
5. `scripts/quality-report.ts` 로 집계 (미구현, 샘플 평가 착수 후 작성)

## 점수 인플레이션 방지

- 평가자 2명의 크로스체크: 축별 차이 ≥ 2점이면 제3자 조정
- 같은 평가자가 동일 응답을 2주 후 재평가 시 차이 ≥ 1점 평균이면 rubric 재교육
- 점수 분포가 과도하게 특정 점수에 몰리면 (예: 전부 3점) rubric 용어 재정의
